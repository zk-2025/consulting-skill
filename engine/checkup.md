# 体检模式引擎

> **解决什么问题**：用户没有具体问题，但想用一套框架把业务拆一遍，看看哪里有问题。
> **什么时候用**：用户选"体检"模式时启动。
> **联动文件**：体检完成后 → `output-format.md` 输出诊断报告 → `report.md` 生成报告

---

## 体检流程状态机

```
CHECKUP-INIT → SCANNING → DIAGNOSING → REPORTING
```

| 当前状态 | 触发条件 | 下一状态 | 动作 |
|----------|----------|----------|------|
| **CHECKUP-INIT** | 用户选择体检模式 | SCANNING | 说明体检流程，询问"快速体检还是完整体检？" |
| **SCANNING** | 当前维度信息收集完成 | SCANNING | 追问下一维度 |
| **SCANNING** | 所有维度收集完成 | DIAGNOSING | 进入框架对照诊断 |
| **DIAGNOSING** | 逐层诊断完成 | REPORTING | 生成诊断报告 |

### 体检模式选择

| 模式 | 每个维度问题数 | 总轮次 | 适用场景 |
|------|---------------|--------|----------|
| **快速体检** | 1个最关键问题 | 5轮 | 时间紧、想快速了解全局 |
| **完整体检** | 3个问题 | 15轮 | 需要深度诊断、有充足时间 |

> 快速体检发现异常维度后，可单独深入追问该维度，无需全部展开。

## Phase A：企业全景扫描（5个维度）

每个维度用追问收集信息，每次只问1个问题（规则见 SKILL.md 回复节奏控制）。

### 维度1：战略层

**快速体检关键问题**（只问第1个）：
1. "你们公司现在的使命/愿景是什么？能一句话说清楚吗？" ⭐
2. "你目前同时在做几个大方向？如果只能保一个，保哪个？"
3. "你的核心竞争对手是谁？你和他们的差异在哪？"

**信息收集模板**：
```json
{
  "使命愿景": "",
  "当前战略方向": [],
  "竞争格局": {
    "主要竞品": "",
    "差异化": ""
  }
}
```

### 维度2：运营层

**快速体检关键问题**（只问第1个）：
1. "你们的核心业务流程中，哪个环节最慢/最常出错？" ⭐
2. "你们的核心业务流程是什么？从获客到交付经过哪些环节？"
3. "有没有固定的执行节奏？（周会/月会/季度复盘）"

**信息收集模板**：
```json
{
  "核心流程": "",
  "瓶颈环节": "",
  "执行节奏": "",
  "关键指标": []
}
```

### 维度3：组织层

**快速体检关键问题**（只问第1个）：
1. "每个人清楚本季度最重要的目标是什么吗？他们能复述出来吗？" ⭐
2. "团队有多少人？结构是怎样的？"
3. "目标是怎么对齐的？有没有OKR之类的机制？"

**信息收集模板**：
```json
{
  "团队规模": "",
  "组织结构": "",
  "目标对齐方式": "",
  "人才梯队状况": ""
}
```

### 维度4：市场层

**快速体检关键问题**（只问第1个）：
1. "你们最核心的客户是谁？他们为什么选你们而不选竞品？" ⭐
2. "获客渠道有哪些？获客成本大概多少？"
3. "定价策略是什么？基于成本、竞争还是价值？"

**信息收集模板**：
```json
{
  "目标客户": "",
  "价值主张": "",
  "获客渠道": [],
  "获客成本": "",
  "定价策略": ""
}
```

### 维度5：增长层

**快速体检关键问题**（只问第1个）：
1. "用户留存率大概多少？从用户知道你到付费，哪一步流失最多？" ⭐
2. "行业平均留存率呢？"
3. "用户会主动推荐你们吗？推荐机制是什么？"

**信息收集模板**：
```json
{
  "留存率": "",
  "行业平均留存": "",
  "漏斗断点": "",
  "推荐机制": ""
}
```

## Phase B：框架对照诊断

逐层对照公理和模块框架，识别错位/缺失/矛盾。

### 诊断对照表

| 维度 | 对照框架 | 诊断要点 | 常见问题 |
|------|----------|----------|----------|
| 战略层 | 好战略三要素（调查分析→指导方针→连贯性活动） | 是否有"空话"？是否错把目标当战略？是否同时追多个目标？ | ❌ 战略是空话 ❌ 目标当战略 ❌ 多目标无优先级 |
| 战略层（毛选） | 调查研究法 + 矛盾分析法 + 集中优势兵力 | 决策是否基于调查？是否抓住了主要矛盾？资源是否集中在关键突破点？ | ❌ 拍脑袋决策 ❌ 眉毛胡子一把抓 ❌ 资源平均分配 |
| 运营层 | 四维诊断（目标对齐/执行机制/流程效率/指标反馈） | 有没有执行节奏？瓶颈在哪？指标是否驱动决策？ | ❌ 没有检查机制 ❌ 瓶颈未识别 ❌ 指标是装饰 |
| 运营层（毛选） | 实践认识循环 | 方案是否经实践验证？有没有从实践中修正认知？ | ❌ PPT方案未验证 ❌ 只认逻辑不认事实 |
| 组织层 | OKR + 领导梯队 | 目标是否对齐？领导力是否转型？梯队是否断层？ | ❌ 目标未对齐 ❌ 虚假对齐 ❌ 梯队断层 |
| 组织层（毛选） | 群众路线法 | 目标是自上而下命令还是自下而上汇聚？团队是否理解并认同？ | ❌ 目标只有下达没有汇聚 ❌ 团队被动接受 |
| 市场层 | 5C分析 + CBBE | 客户是否聚焦？价值主张是否清晰？品牌资产在哪一层？ | ❌ 客户不聚焦 ❌ 差异化不清 ❌ 品牌停在认知层 |
| 增长层 | AARRR + STEPPS | 留存是否健康？漏斗断点在哪？有没有内生传播机制？ | ❌ 漏桶未修先获客 ❌ 没有行为剩余 ❌ 缺社交货币 |
| 增长层（毛选） | 持久战三阶段 | 当前处于防御、相持还是反攻阶段？策略是否匹配阶段？ | ❌ 防御期用进攻策略 ❌ 速胜论 ❌ 没有阶段判断 |

### 公理违反检测

逐条检查公理是否被违反。按优先级分三档：🔴必查、🟡看情况、🟢低频。

| 优先级 | 公理 | 检测方法 | 违反信号 |
|--------|------|----------|----------|
| 🔴 | 公理1：问题在被精确描述之前不存在解决方案 | 问"你的核心问题是什么？" | 回答模糊/笼统 |
| 🔴 | 公理3：战略的本质是取舍 | 问"你同时在做几个方向？" | >3个且无优先级 |
| 🔴 | 公理8：执行力不是意愿问题是系统问题 | 检查目标对齐和机制 | 归因于"人不行" |
| 🔴 | 公理11：很多问题不是问题，是不愿尝试、不愿承认、不愿执行 | 问"你试过了吗？" | 问"行不行"但不去试；绕着真问题走 |
| 🟡 | 公理2：好战略包含三个必须要素 | 检查战略是否有调查分析+指导方针+连贯活动 | 缺少任一要素 |
| 🟡 | 公理5：流量≠收入，品牌≠规模 | 检查是否在盲目追流量 | 有流量无转化 |
| 🟡 | 公理6：先修漏桶再拉新水 | 检查留存率 | 留存差但疯狂获客 |
| 🟡 | 公理9：传播是产品设计的延伸 | 检查产品是否有内生传播机制 | 传播全靠广告投放 |
| 🟡 | 公理10：品牌资产是结果不是目标 | 检查品牌建设是否从产品出发 | 先做品牌再做产品 |
| 🟡 | 公理12：精准提问是解决问题的起点 | 检查问题定义是否可验证 | 问题描述用大词/无法量化 |
| 🟡 | 公理13：最好的零件是没有零件 | 检查是否在优化不该存在的东西 | 功能/流程堆砌，无人问"该不该删" |
| 🟢 | 公理4：商业模式是独立于人的客观存在 | 检查是否依赖某个关键人 | 核心流程依赖个人 |
| 🟢 | 公理7：组织是为战略服务的 | 检查组织结构是否匹配战略 | 组织先于战略设计 |
| 🟢 | 公理14：人数不能弥补人才 | 检查团队扩张是否带来等比例产出 | 人多但决策慢、产出低 |
| 🟢 | 公理15：如果设计需要很长时间来建造，那就是错误的设计 | 检查项目周期是否异常长 | 造不出来说明设计没想透 |
| 🟢 | 公理16：制造机器的机器比机器本身难一万倍 | 检查是否有量产能力 | 有原型但无法规模化交付 |
| 🟡 | 公理17：聪明人最常见的错误是优化不该存在的东西 | 检查优化方向是否对 | 在错误方向上追求效率 |
| 🔴 | 公理18：调查就是解决问题 | 问"这个结论是基于什么调查得出的？" | 拍脑袋决策；没有数据就出方案；"我觉得"代替"我发现" |
| 🔴 | 公理19：抓主要矛盾，一切问题迎刃而解 | 问"你现在最核心的矛盾是什么？次要矛盾有哪些？" | 眉毛胡子一把抓；所有问题同等优先级；找不到中心 |
| 🔴 | 公理20：集中优势兵力，各个歼灭敌人 | 检查资源分配是否聚焦关键突破点 | 资源平均分配到所有方向；多点同时出击；每战不能形成局部优势 |
| 🟡 | 公理21：持久战的胜利属于有耐心、有策略的一方 | 问"你判断当前处于防御、相持还是反攻阶段？" | 防御期用进攻策略；速胜论或亡国论；没有阶段判断就行动 |
| 🔴 | 公理22：实践是检验真理的唯一标准 | 问"这个方案实践验证过了吗？有实物吗？" | 只有PPT没有实物；逻辑完美但未验证；方案停留在纸面 |
| 🟡 | 公理23：不同质的矛盾用不同质的方法解决 | 检查是否对不同类型问题用了不同解法 | 套用一个模板解决所有问题；一刀切；不区分问题性质 |

## Phase C：诊断报告输出（体检模式对话内输出；用户说"出报告"时转 `engine/report.md` 生成正式文件）

```markdown
## 体检诊断报告

### 一句话核心发现
{最严重的公理违反或框架缺失}

### 问题清单（按优先级）
1. 🔴 {最严重的问题} — {违反哪条公理/缺失哪个框架} — {影响}
2. 🟡 {次严重的问题} — {证据} — {影响}
3. 🟢 {需关注的问题} — {证据} — {影响}

### 行动路线图
1. 第1步：{做什么} — {为什么先做这个}
2. 第2步：{做什么} — {依赖第1步的什么结果}
3. 第3步：{做什么} — {依赖第2步的什么结果}

### 关键验证信号
- 做对的信号：{什么现象说明方向对了}
- 做错的信号：{什么现象说明需要调整}

### 建议深入诊断的模块
- {模块名}：{为什么需要深入}
```

## 体检节奏控制

1. **每个维度追问不超过3个问题**，避免用户疲劳
2. **5个维度之间给用户喘息**："战略层的信息收到了。接下来看看运营层——"
3. **诊断结论按严重程度排序**，最严重的放最前面
4. **行动路线图最多3步**，不要给用户太多选择
